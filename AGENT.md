# AGENT.md

本文件是本项目给 AI agent / coding agent 使用的本地执行规则。它补充并固化当前项目约定；若与更高层系统规则冲突，以更高层规则为准。

## 1. 项目背景

`家校签收通` 是面向学校的家长通知签收 MVP。当前形态是：

- Markdown 文档：需求、实施决策和验收口径的源头。
- `web/`：Next/React 正式前端工程。
- `server/`：Express + SQLite MVP 后端，CommonJS 风格。

核心目标是让学校发布通知、班主任转发并跟进、家长在线绑定和手写签收，最后形成可追溯的 Excel/PDF/zip 归档。

## 2. 源头与回写规则

- Markdown 是可执行的事实源头。
- HTML 是视觉/交互 artifact 层，不单独承载最终决策。
- 前端、接口或实现中出现新的产品决策，必须回写到 Markdown：
  - 面向开源用户的产品/流程/状态说明：更新 `README.md` 或相应公开文档；完整内部 PRD 留在本地开发仓库。
  - 部署、端口、备份、安全配置：更新 `02-实施配置与部署约定.md`。
  - 前端入口、页面结构和本地开发约定：更新 `03-前端开发说明.md`。
  - 面向项目入口和当前进展：必要时更新 `README.md`。
- 不要只改前端、接口或后端代码而留下文档过期。

## 3. 工作方式

- 动手前先明确假设；如果存在多个合理解释，先指出差异。
- 优先做最小可验证改动，不添加未被要求的功能、抽象或配置项。
- 只改与任务直接相关的文件；不要顺手重构、改格式或清理无关代码。
- 若发现无关问题，可以在回复中说明，但不要擅自删除或重写。
- 每个多步任务都要有可验证成功标准，例如“接口返回正确”“截图可复现”“文档与实现一致”。
- 对现有风格保持一致：文档用中文说明，后端保持 CommonJS，前端保持当前 Next/React 结构。

## 4. MVP 边界

除非用户明确要求并同步更新需求文档，不要突破以下边界：

- 家长端不做账号体系、个人中心、手机号验证码、公众号强绑定。
- MVP 只做一种通知类型：所有通知都要求家长手写签名。
- 学生导入模板只包含 `学生姓名` 和 `班内序号`。
- 不做全校大盘看板，学校端以班级进度表和导出入口为主。
- 不做短信、微信自动发送；提醒以复制话术和人工跟进为主。
- PDF 归档采用 HTML/富文本模板转 PDF，不做原始 PDF 坐标签章。
- 班级批量下载生成 zip，默认包含本班已签学生 PDF。

## 5. 安全与数据边界

- 绝不写入 SSH 私钥、密码、Token、Cookie、数据库密码、真实密钥或任何凭据。
- 不要把秘密放进 Markdown、代码、脚本、环境变量模板、日志或截图。
- 任何曾在对话、本地文件或日志中出现过的 SSH 私钥不得复用到项目文件；真实试点前应更换密钥或使用独立部署用户。
- IP + HTTP 端口只允许假数据、脱敏数据或内部流程演示。
- 真实学生、家长、签名图片、IP、User-Agent 等数据必须通过域名 + HTTPS 使用。
- 签名图片、PDF、zip 下载必须经过鉴权，不允许仅靠静态路径暴露。
- 避免扩大个人信息采集范围；MVP 不收身份证、人脸、家庭住址、精确定位。

## 6. 前端约定

- 正式前端入口目录是 `web/`。
- 不要再新增 `prototype/` 目录作为运行时代码入口。
- 后端未启动时，允许使用 `web/lib/demo-data.ts` 的假数据作为本地演示兜底。
- 接入真实接口时，应优先读取后端数据；失败时保留清晰的降级状态。
- 视觉方向是“学校教务台账”：克制、密集、可扫读，适合老师和学校管理员反复处理。
- 家长端优先窄屏体验，突出绑定、隐私告知、确认声明和手写签名动作。
- 页面组件按正式视图组织在 `web/components/views`，不要使用“原型板总览”作为正式入口。

## 7. 后端约定

- 后端入口：`server/server.js`。
- 本地端口默认 `8088`，可通过 `PORT` 环境变量覆盖。
- 数据目录：`server/data`，上传目录：`server/data/uploads`，导出目录：`server/data/exports`。
- 数据库为 SQLite，文件默认 `server/data/jiaxiao-sign.sqlite`。
- 保持当前 Express + better-sqlite3 + pdfkit + archiver 的简单 MVP 结构。
- 本地开发 demo 通道 `x-demo-user` 仅用于非生产环境，不得扩大到生产逻辑。
- 改动鉴权、导出、签名、数据保存逻辑时，必须额外检查权限范围和真实数据安全边界。

## 8. 常用命令

启动后端：

```powershell
cd server
npm install
npm run start
```

健康检查：

```powershell
Invoke-RestMethod http://localhost:8088/api/health
```

启动前端：

```powershell
cd web
corepack pnpm install
corepack pnpm dev
```

构建前端：

```powershell
cd web
corepack pnpm build
```

## 9. 验证要求

- 文档改动：检查 README 和三份编号文档之间是否一致。
- 前端改动：运行 `corepack pnpm build`；需要时启动前端并打开相关页面。
- 后端改动：启动服务，至少检查 `/api/health`，并手动或脚本验证被改接口。
- 导出/PDF/zip 改动：验证文件生成、下载鉴权、文件名和权限范围。
- 涉及真实试点前的改动：复核 HTTPS、密钥轮换、备份恢复、隐私告知和数据保留周期。

## 10. 不要触碰

- 不要编辑 `server/node_modules/`。
- 不要编辑 `web/node_modules/`。
- 不要把 `server/data/` 中的真实或试点数据提交、复制到文档或截图说明中。
- 不要把日志里的敏感内容写回项目文档。
- 不要为了微小任务重写大文件或重排已有文档结构。
